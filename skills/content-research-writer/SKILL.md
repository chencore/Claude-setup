# 内容研究写作器

此技能作为您的写作伙伴，帮助您研究、概述、起草和润色内容，同时保持您独特的声音和风格。

## 何时使用此技能

- 编写博客文章、文章或新闻通讯
- 创建教育内容或教程
- 起草思想领导力文章
- 研究和撰写案例研究
- 生成带有来源的技术文档
- 使用适当的引用和参考文献进行写作
- 改进钩子和引言
- 在写作过程中获得分节反馈

## 此技能的功能

1. **协作式 outlining**：帮助您将想法结构化为连贯的大纲
2. **研究协助**：查找相关信息并添加引用
3. **钩子改进**：加强您的开场白以吸引注意力
4. **节反馈**：在您写作时审阅每个节
5. **声音保留**：保持您的写作风格和语气
6. **引用管理**：适当地添加和格式化参考文献
7. **迭代式改进**：通过多个草稿帮助您改进

## 如何使用

### 设置您的写作环境

为您的文章创建专用文件夹：
```
mkdir ~/writing/my-article-title
cd ~/writing/my-article-title
```

创建您的草稿文件：
```
touch article-draft.md
```

从此目录打开 Claude Code 并开始写作。

### 基本工作流程

1. **从大纲开始**：
```
帮助我为关于 [主题] 的文章创建大纲
```

2. **研究和添加引用**：
```
研究 [特定主题] 并为我的大纲添加引用
```

3. **改进钩子**：
```
这是我的引言。帮助使钩子更具吸引力。
```

4. **获得节反馈**：
```
我刚完成"为什么这很重要"部分。审阅它并提供反馈。
```

5. **润色和完善**：
```
审阅完整草稿的流程、清晰度和一致性。
```

## 指令

当用户请求写作协助时：

1. **理解写作项目**

   询问澄清性问题：
   - 主题和主要论点是什么？
   - 目标受众是谁？
   - 期望的长度/格式是什么？
   - 您的目标是什么？（教育、说服、娱乐、解释）
   - 有任何现有研究或来源要包含吗？
   - 您的写作风格是什么？（正式、对话式、技术性）

2. **协作式 outlining**

   帮助结构化内容：

   ```markdown
   # 文章大纲：[标题]

   ## 钩子
   - [开场白/故事/统计数据]
   - [为什么读者应该关心]

   ## 引言
   - 上下文和背景
   - 问题陈述
   - 本文内容概述

   ## 主要部分

   ### 第1节：[标题]
   - 关键点 A
   - 关键点 B
   - 示例/证据
   - [需要研究：特定主题]

   ### 第2节：[标题]
   - 关键点 C
   - 关键点 D
   - 需要的数据/引用

   ### 第3节：[标题]
   - 关键点 E
   - 反对论点
   - 解决方案

   ## 结论
   - 主要要点总结
   - 呼吁行动
   - 最后的想法

   ## 研究待办事项
   - [ ] 查找 [主题] 的数据
   - [ ] 获取 [概念] 的示例
   - [ ] 为 [主张] 来源引用
   ```

   **迭代大纲**：
   - 根据反馈调整
   - 确保逻辑流程
   - 识别研究空白
   - 标记需要深入探讨的节

3. **进行研究**

   当用户请求研究某个主题时：

   - 搜索相关信息
   - 查找可靠来源
   - 提取关键事实、引言和数据
   - 以请求的格式添加引用

   示例输出：
   ```markdown
   ## 研究：AI 对生产力的影响

   关键发现：

   1. **生产力提升**：研究表明内容创建任务节省了 40% 的时间 [1]

   2. **采用率**：67% 的知识工作者每周使用 AI 工具 [2]

   3. **专家引言**："AI 增强而非取代人类创造力" - Jane Smith 博士，麻省理工学院 [3]

   引用：
   [1] McKinsey Global Institute. (2024). "生成式 AI 的经济潜力"
   [2] Stack Overflow 开发人员调查 (2024)
   [3] Smith, J. (2024). MIT 科技评论采访

   添加到第 2 节下的大纲中。
   ```

4. **改进钩子**

   当用户分享引言时，分析和加强：

   **当前钩子分析**：
   - 有效之处：[积极要素]
   - 可以更强之处：[改进领域]
   - 情感影响：[当前 vs 潜力]

   **建议替代方案**：

   选项 1：[大胆声明]
   > [示例]
   *为什么有效：[解释]*

   选项 2：[个人故事]
   > [示例]
   *为什么有效：[解释]*

   选项 3：[惊人数据]
   > [示例]
   *为什么有效：[解释]*

   **钩子问题**：
   - 它是否创造好奇心？
   - 它是否承诺价值？
   - 是否足够具体？
   - 它是否匹配受众？

5. **提供分节反馈**

   当用户写作每个节时，审阅：

   ```markdown
   # 反馈：[节名称]

   ## 有效之处 ✓
   - [优势 1]
   - [优势 2]
   - [优势 3]

   ## 改进建议

   ### 清晰度
   - [具体问题] → [建议修复]
   - [复杂句子] → [更简单的替代]

   ### 流程
   - [过渡问题] → [更好的连接]
   - [段落顺序] → [建议重新排序]

   ### 证据
   - [需要支持的主张] → [添加引用或示例]
   - [泛泛陈述] → [更具体]

   ### 风格
   - [语气不一致] → [更好地匹配您的声音]
   - [用词选择] → [更强的替代]

   ## 具体行编辑

   原文：
   > [草稿中的准确引用]

   建议：
   > [改进版本]

   原因：[解释]

   ## 考虑的问题
   - [发人深省的问题 1]
   - [发人深省的问题 2]

   准备进入下一节！
   ```

6. **保留作者声音**

   重要原则：

   - **学习他们的风格**：阅读现有写作样本
   - **建议，不要替代**：提供选项，而不是指令
   - **匹配语气**：正式、随意、技术性、友好
   - **尊重选择**：如果他们喜欢自己的版本，支持它
   - **增强，而不是覆盖**：使他们的写作更好，而不是不同

   定期询问：
   - "这听起来像您吗？"
   - "这是正确的语气吗？"
   - "我应该更多/更少 [正式/随意/技术性] 吗？"

7. **引用管理**

   根据用户偏好处理参考文献：

   **内联引用**：
   ```markdown
   研究显示 40% 的生产力提升（麦肯锡，2024）。
   ```

   **编号引用**：
   ```markdown
   研究显示 40% 的生产力提升 [1]。

   [1] McKinsey Global Institute. (2024)...
   ```

   **脚注样式**：
   ```markdown
   研究显示 40% 的生产力提升^1

   ^1: McKinsey Global Institute. (2024)...
   ```

   维护运行引用列表：
   ```markdown
   ## 参考文献

   1. 作者。（年份）。"标题"。出版物。
   2. 作者。（年份）。"标题"。出版物。
   ...
   ```

8. **最终审阅和润色**

   当草稿完成时，提供全面反馈：

   ```markdown
   # 完整草稿审阅

   ## 总体评估

   **优势**：
   - [主要优势 1]
   - [主要优势 2]
   - [主要优势 3]

   **影响**：[整体效果评估]

   ## 结构与流程
   - [关于组织的评论]
   - [过渡质量]
   - [节奏评估]

   ## 内容质量
   - [论点强度]
   - [证据充分性]
   - [示例有效性]

   ## 技术质量
   - 语法和机械：[评估]
   - 一致性：[评估]
   - 引用：[完整性检查]

   ## 可读性
   - 清晰度评分：[评估]
   - 句子多样性：[评估]
   - 段落长度：[评估]

   ## 最终润色建议

   1. **引言**：[具体改进]
   2. **主体**：[具体改进]
   3. **结论**：[具体改进]
   4. **标题**：[如果需要，提供选项]

   ## 发布前检查清单
   - [ ] 所有主张都有来源
   - [ ] 引用格式正确
   - [ ] 示例清晰
   - [ ] 过渡流畅
   - [ ] 包含呼吁行动
   - [ ] 校对拼写错误

   准备发布！🚀
   ```

## 示例

### 示例 1：Teresa Torres 的工作流程

**用户**："我正在写一篇关于持续发现的文章。帮助我创建一个大纲。"

**流程**：
1. 协作大纲结构
2. 识别研究需求
3. 用户开始写作引言
4. 改进钩子
5. 用户写作每个节
6. 每节后提供反馈
7. 进行研究和添加引用
8. 完整草稿的最终审阅
9. 润色并为发布准备

**结果**：经过充分研究、正确引用的文章，以 Teresa 的声音写作，具有强大的结构和流程。

### 示例 2：研究型文章

**用户**："我正在写 AI 对产品管理的影响。帮助我找到当前的数据和示例。"

**输出**：
```markdown
## 已编译研究

### 最近数据
- Gartner：到 2025 年，80% 的产品经理将使用 AI 工具 [1]
- 调查：45% 报告更快的功能发布 [2]
- 案例研究：Airbnb 的 AI 辅助优先级排序 [3]

### 专家引言
- "AI 放大而非取代产品经理判断" - Marty Cagan
- [其他带引用的引言]

### 真实示例
1. **公司 A**：使用 AI 进行用户研究综合
   - 结果：节省 60% 时间
   - 来源：[引用]

2. **公司 B**：AI 驱动的路线图分析
   - 结果：更好的利益相关者对齐
   - 来源：[引用]

所有已添加到您的大纲中，具有适当的引用。
```

### 示例 3：钩子改进

**用户的原始钩子**：
> "产品管理因 AI 而改变。在本文中，我将讨论 AI 影响产品经理的一些方式。"

**改进选项**：

**选项 1（数据驱动）**：
> "上个月，我要求 AI 分析 500 次客户访谈。它用了 30 分钟而不是 3 周。产品管理将永远改变。"

**选项 2（问题）**：
> "如果您能在早晨咖啡之前与每个客户交谈、阅读每篇评论并分析每张支持工单会怎样？"

**选项 3（故事）**：
> "Sarah 花了两周时间构建了错误的功能。不是因为不了解她的用户，而是因为她无法足够快地处理数百次访谈以发现模式。"

### 示例 4：节反馈

**用户**："刚完成我的'常见错误'部分。有什么想法？"

**回应**：
```markdown
# 反馈：常见错误部分

## 有效之处 ✓
- 强大的示例（Slack 通知故事）
- 清晰的结构（3 个错误，干净的布局）
- 每个错误的实用建议

## 建议

### 使其更具体
您的第二点说"糟糕的优先级排序"，但可以更强：

当前：
> "许多团队通过专注于功能而不是成果而优先级排序错误。"

建议：
> "我曾看过一个团队在一个季度内发布了 14 个功能，但没有移动任何关键指标。他们优先考虑了活动而不是进展。"

### 添加数据
第三个错误将从证据中受益：
> "[添加引用]：研究表明没有定期用户联系功能的团队构建未使用功能的可能性是 3 倍 [需要来源]"

### 流程改进
考虑重新排序：错误 3 → 错误 2 → 错误 1
这从小影响到大影响构建。

准备进入下一节！
```

## 写作工作流程

### 博客文章工作流程
1. 一起制定大纲
2. 研究关键点
3. 写引言 → 获得反馈
4. 写主体节 → 每节反馈
5. 写结论 → 最终审阅
6. 润色和编辑

### 新闻通讯工作流程
1. 讨论钩子想法
2. 快速大纲（较短格式）
3. 一次会议中起草
4. 审阅清晰度和链接
5. 快速润色

### 技术教程工作流程
1. 概述步骤
2. 编写代码示例
3. 添加解释
4. 测试说明
5. 添加故障排除部分
6. 准确性的最终审阅

### 思想领导力工作流程
1. 独特角度的头脑风暴
2. 研究现有观点
3. 发展您的论点
4. 以强烈的 POV 写作
5. 添加支持证据
6. 设计令人信服的结论

## 专业提示

1. **在 VS Code 中工作**：比网络 Claude 更适合长篇写作
2. **一次一个节**：增量获得反馈
3. **单独保存研究**：保留 research.md 文件
4. **版本化您的草稿**：article-v1.md、article-v2.md 等
5. **大声朗读**：使用反馈识别笨拙的句子
6. **设定截止日期**："我想今天完成草稿"
7. **休息**：写作、获得反馈、暂停、修改

## 文件组织

写作项目建议结构：

```
~/writing/article-name/
├── outline.md          # 您的大纲
├── research.md         # 所有研究和引用
├── draft-v1.md         # 第一稿
├── draft-v2.md         # 修改稿
├── final.md            # 发布就绪
├── feedback.md         # 收集的反馈
└── sources/            # 参考材料
    ├── study1.pdf
    └── article2.md
```

## 最佳实践

### 对于研究
- 引用前验证来源
- 可能时使用最新数据
- 平衡不同观点
- 链接到原始来源

### 对于反馈
- 明确说明您想要什么："这太技术性了吗？"
- 分享您的担忧："我担心这部分拖沓"
- 提问："这逻辑流畅吗？"
- 请求替代方案："有什么其他方式解释这个？"

### 对于声音
- 分享您的写作示例
- 指定语气偏好
- 指出良好的匹配："这听起来像我！"
- 标记不匹配："对我的风格来说太正式了"
